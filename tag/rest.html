Tutorial Django-REST-Framework - Parte 2

Sat 18 January 2014 | django / django-packages / REST | Link permanente para este artigo

Tutorial 2: Requests e Responses

A partir deste ponto iremos cobrir o core do framework REST. Vamos introduzir algumas partes essenciais da construção.

Objetos Request.

O framework REST introduz um objeto Reuquest que extende o HttpRequest normal, e fornece maior flexibilidade no tratamento do request. A principal funcionalidade do objeto Request e o atributo request.DATA, o qual é similar ao request.POST, mas mais funcional para trabalhar com uma API Web.

request.POST  # Only handles form data.  Only works for 'POST' method.
request.DATA  # Handles arbitrary data.  Works for 'POST', 'PUT' and 'PATCH' methods.

Objetos Response

o framework REST também introduz um objeto Response, o qual é um tipo de TemplateResponse que trata um conteúdo não renderizado e usa o mecanismo de content negotiation para determinar o tipo correto de conteúdo para retornar ao cliente.

return Response(data)  # Renders to content type as requested by the client.

Códigos de Status.

O uso de códigos de status HTTP nas suas views nem sempre faz com que a leitura seja clara, e é fácil não notar se você leu um código de erro errado. O framework REST fornece identificadores mais explícitos para cada código de status, tal como um HTP_404_BAD_REQUEST no módulo de status. É uma boa idéia usar isso em tudo ao invés de identificadores numéricos.

Wrapping views

*Nota do tradutor: Wrapping é um padrão no desenvolvimento de software - ver wrapper function. a tradução do título seria, "envolvendo funções do tipo 'view'". em python, é equivalente ao uso de 'decorators'. O termo "wrapper"

O framework REST fornece dois wrappers que você pode usar para escrever views da API.

  1. O decorator @api_view, para trabalhar com views como funções.
  2. A Classe APIView, para trabalhar com views baseadas em classes (CBS - Django Class Based Views).

Estes wrappers fornecem partes de algumas funcionalidades tal como ter certeza que você recebeu instâncias de Request na sua view, e adicionando context aos objetos Response então o content negotiation pode ser feito.

Os wrappers também fornecem comportamento tal como retornar uma resposta 405 Method Not Allowed quando for necessário, e tratar qualquer exceção do tipo ParserError que pode ocorrer quando uma entrada mal formada no request.DATA é acessada.

Juntando tudo.

Okay, vamos em frente e usar estes novos componentes para escrever algumas views.

Nós não precisamos mais da nossa classe JSONResponse no views.py, então vá em frente e delete isso. Uma vez feito isso, podemos aos poucos, começar nosso refactoring.

!python
from rest_framework import status
from rest_framework.decorators import api_view
from rest_framework.response import Response
from snippets.models import Snippet
from snippets.serializers import SnippetSerializer


@api_view(['GET', 'POST'])
def snippet_list(request):
    """
    List all snippets, or create a new snippet.
    """
    if request.method == 'GET':
        snippets = Snippet.objects.all()
        serializer = SnippetSerializer(snippets, many=True)
        return Response(serializer.data)

    elif request.method == 'POST':
        serializer = SnippetSerializer(data=request.DATA)
        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=status.HTTP_201_CREATED)
        return Response(serializer.errors, status=status.HTTP_400_BAD_REQUEST)

Nossa view de instância é uma melhoria se comparada ao exemplo anterior. É um pouco mais concisa, e o código é um pouco mais parecido com o que trabalhamos na API de Forms . Também estamos usando códigos de status nomeados, o que torna o significado da resposta um pouco mais óbvio.

Aqui a view para um snippet individual, no módulo views.py.

!python @api_view(['GET', 'PUT', 'DELETE']) def snippet_detail(request, pk): """ Retrieve, update or delete a snippet instance. """ try: snippet = Snippet.objects.get(pk=pk) except Snippet.DoesNotExist: return Response(status=status.HTTP_404_NOT_FOUND)

if request.method == 'GET':
    serializer = SnippetSerializer(snippet)
    return Response(serializer.data)

elif request.method == 'PUT':
    serializer = SnippetSerializer(snippet, data=request.DATA)
    if serializer.is_valid():
        serializer.save()
        return Response(serializer.data)
    return Response(serializer.errors, status=status.HTTP_400_BAD_REQUEST)

elif request.method == 'DELETE':
    snippet.delete()
    return Response(status=status.HTTP_204_NO_CONTENT)

Isso tudo deve parecer muito familiar - Não é muito diferente de trabalhar com as views normais do Django.

Note que não estamos mais amarrando nossos requests ou responses a um determinado tipo de conteúdo. request.DATA pode tratar requests json, mas pode também tratar também yaml e outros formatos. Da mesma maneira, estamos retornando objetos reponse com dados, mas permitindo ao REST framework renderizar o reponse no tipo correto de conteúdo pra nós.

Adicionando sufixos de formatos opcionais a nossas URLs

Para tirar vantagem do fato que nossos reponses não estão mais amarrados a um único tipo de conteúdo, vamos adicionar suporte a sufixos de formatos para noss API. Usar sufixos de formatos nos dá URLs que explicitamente referenciam um dado formato, e significa que nossa API será capaz de tratar URLs tal como http://example.com/api/items/4.json.

Começamos por adicionar o keyword argumento format para ambas as views, como abaixo.

def snippet_list(request, format=None):

e

def snippet_detail(request, pk, format=None):

Agora atualize o arquivo urls.py , para concatenar um conjunto de format_suffix_patterns as URLs existentes.

from django.conf.urls import patterns, url
from rest_framework.urlpatterns import format_suffix_patterns

urlpatterns = patterns('snippets.views',
    url(r'^snippets/$', 'snippet_list'),
    url(r'^snippets/(?P<pk>[0-9]+)$', 'snippet_detail'),
)

urlpatterns = format_suffix_patterns(urlpatterns)

Nós não precisamos necessariamente adicionar esses padrões de URLs, mas isso nos dá uma maneira simples e clara de referenciar um formato específico.

Como lhe parece?

Vá em frente e teste a API na linha de comando, como fizemos no tutorial part 1. Tudo está funcionando de maneira similar, embora nós tenhamos alguns bons tratamentos de erros se enviarmos request inválidos.

Nós podemos ter uma lista de snippets como antes.

curl http://127.0.0.1:8000/snippets/

[{"id": 1, "title": "", "code": "foo = \"bar\"\n", "linenos": false, "language": "python", "style": "friendly"}, {"id": 2, "title": "", "code": "print \"hello, world\"\n", "linenos": false, "language": "python", "style": "friendly"}]

Podemos controlar o formato da resposta, ou através do uso do cabeçalho Accept:

curl http://127.0.0.1:8000/snippets/ -H 'Accept: application/json'  # Request JSON
curl http://127.0.0.1:8000/snippets/ -H 'Accept: text/html'         # Request HTML

Ou concatenando o sufixo do formato:

curl http://127.0.0.1:8000/snippets/.json  # JSON suffix
curl http://127.0.0.1:8000/snippets/.api   # Browsable API suffix

Da mesma maneira, podemos controlar o formato dos requests que enviamos, usando o cabeçalho Content-Type.

# POST using form data
curl -X POST http://127.0.0.1:8000/snippets/ -d "code=print 123"

{"id": 3, "title": "", "code": "print 123", "linenos": false, "language": "python", "style": "friendly"}

# POST using JSON
curl -X POST http://127.0.0.1:8000/snippets/ -d '{"code": "print 456"}' -H "Content-Type: application/json"

{"id": 4, "title": "", "code": "print 456", "linenos": true, "language": "python", "style": "friendly"}

Agora abra a API em um navegador Web, visitando http://127.0.0.1:8000/snippets/.

Navegabilidade

Como a API escolhe o tipo de conteúdo do response baseado no request do cliente, isso irá por padrão, retornar uma representação formatada em HTML do recurso quando este recurso é requisitado por um navegador Web. Isso permite a API retornar uma representação HTML totalmente compatível com seu navegador Web.

Ter uma API acessível ao seu navegador Web é um grande ganho de usabilidade, e faz com que o desenvolvimento e uso da sua API seja mais fácil. Isso também diminui dramaticamente a barreira de entrada para outros desenvolvedores que queiram inspecionar e trabalhar com sua API.

Veja o tópico API Navegável para mais informações sobre a característica da API navegável e como customiza-lá.
